## SECTION 2: Spatial Map Logic and User Discrimination

### 2.1 GPS Coordinate Sync (lat/lng lifecycle)

Deep analytical explanation:
- Coordinates are tracked by navigator.geolocation.watchPosition in MapViewInner.
- On each position callback:
  - local state userPos/currentLat/currentLng is updated.
  - Supabase profiles row for current user is updated with last_lat, last_lng, updated_at.
- If ghost mode is active, geolocation tracking is skipped and DB coordinates are wiped on transition off->on.

Canonical code snippet (app/components/map/MapViewInner.tsx):
~~~ts
useEffect(() => {
  if (isGhostMode || !('geolocation' in navigator)) return;

  const watchId = navigator.geolocation.watchPosition(
    async (pos) => {
      const lat = pos.coords.latitude;
      const lng = pos.coords.longitude;
      setUserPos([lat, lng]);
      setCurrentLat(lat);
      setCurrentLng(lng);

      await supabase
        .from('profiles')
        .update({ last_lat: lat, last_lng: lng, updated_at: new Date().toISOString() })
        .eq('id', userId);
    },
    () => {/* permission denied or unavailable */},
    { enableHighAccuracy: true, maximumAge: 5000, timeout: 10000 }
  );

  return () => navigator.geolocation.clearWatch(watchId);
}, [userId, supabase, isGhostMode]);
~~~

Ghost wipe transition (app/components/map/MapViewInner.tsx):
~~~ts
if (!wasGhost && isGhostMode) {
  supabase
    .from('profiles')
    .update({ last_lat: null, last_lng: null })
    .eq('id', userId)
}
~~~

Mobile parity goal:
- Flutter must run continuous position stream with high-accuracy and periodic DB updates to profiles.last_lat/last_lng/updated_at.
- When ghost mode turns true, immediately null last_lat and last_lng on server and stop outbound location stream.
- Local map should preserve last known fallback center while hidden.

### 2.2 get_visible_users RPC and strict <= 1 km boundary

Deep analytical explanation:
- Visibility pipeline has two layers:
  1) Supabase RPC get_visible_users(caller_id, user_lat, user_lng)
  2) TypeScript post-filter in route handler that hard-enforces distance <= 1 km.
- Route-level enforcement computes Haversine distance via calculateDistanceKm and removes entries with distanceKm > 1.
- This creates a strict final contract for mobile response payload even if RPC returns broader rows.
- SQL body for RPC is not present in repository; however repository type contract states function returns friends at any distance plus public strangers within 1 km.

Canonical code snippet (app/api/map/visible/route.ts):
~~~ts
const { data, error } = await auth.supabase.rpc('get_visible_users', {
  caller_id: auth.user.id,
  user_lat: lat,
  user_lng: lng,
});

const visibleUsers = ((data ?? []) as VisibleUser[])
  .map((row) => {
    const rowLat = toNumberOrNull(row.last_lat);
    const rowLng = toNumberOrNull(row.last_lng);

    if (rowLat === null || rowLng === null) return null;

    const distanceKm = calculateDistanceKm(lat, lng, rowLat, rowLng);
    if (distanceKm > 1) return null;

    return { ... };
  })
  .filter((row): row is VisibleUserResponse => row !== null);
~~~

Canonical distance math (app/api/_lib/mobile-rest.ts):
~~~ts
export function calculateDistanceKm(fromLat: number, fromLng: number, toLat: number, toLng: number): number {
  const earthRadiusKm = 6_371;
  const toRadians = (value: number) => (value * Math.PI) / 180;

  const deltaLat = toRadians(toLat - fromLat);
  const deltaLng = toRadians(toLng - fromLng);
  const lat1 = toRadians(fromLat);
  const lat2 = toRadians(toLat);

  const a =
    Math.sin(deltaLat / 2) * Math.sin(deltaLat / 2) +
    Math.sin(deltaLng / 2) * Math.sin(deltaLng / 2) * Math.cos(lat1) * Math.cos(lat2);
  const c = 2 * Math.atan2(Math.sqrt(a), Math.sqrt(1 - a));

  return earthRadiusKm * c;
}
~~~

Canonical repository contract (utils/supabase/types.ts):
~~~ts
/**
 * Row returned by the get_visible_users(caller_id, user_lat, user_lng) RPC.
 * The function returns friends (any distance) + public strangers within 1 km.
 */
~~~

Mobile parity goal:
- Flutter map fetch must call RPC with caller_id, user_lat, user_lng.
- Apply post-filter using identical Haversine formula and cutoff strictly distanceKm > 1 rejected.
- Never trust only UI filtering; enforce in data layer before rendering and before badge counts.

### 3.3 Strict pruning rules: 3 hours and max 10 messages

Deep analytical explanation:
- Pruning is enforced in multiple layers:
  1) Client load phase: old records filtered out by 3-hour threshold.
  2) Realtime insert phase: stale inserts ignored if older than threshold.
  3) In-memory retention cap: every append sliced to last 10 using slice(-MAX_MSG).
  4) API history route for mobile/global feed: DB gte(cutoff), safety filter, then slice(-10).
- Repository handoff also states DB trigger deletes messages after 10 or 3 hours; trigger SQL body is not present in codebase here.

Canonical snippets:

Constants and load filter (app/components/chat/ChatPanel.tsx):
~~~ts
const MAX_MSG = 10;

const threshold = Date.now() - 3 * 60 * 60 * 1000;
const filtered = (data as DBMessage[]).filter((m) => new Date(m.created_at).getTime() > threshold);
~~~

Realtime stale guard and cap (app/components/chat/ChatPanel.tsx):
~~~ts
const threshold = Date.now() - 3 * 60 * 60 * 1000;
if (new Date(m.created_at).getTime() <= threshold) return;
setMessages((prev) => [...prev, nextMessage].slice(-MAX_MSG));
~~~

Optimistic append cap (app/components/chat/ChatPanel.tsx):
~~~ts
setMessages((prev) => [...prev, opt].slice(-MAX_MSG));
~~~

Render-time visibility guard (app/components/chat/ChatPanel.tsx):
~~~ts
const threshold = Date.now() - 3 * 60 * 60 * 1000;
const visibleMessages = messages.filter((m) => new Date(m.createdAt).getTime() > threshold);
~~~

Mobile history route cap and age filter (app/api/chat/history/route.ts):
~~~ts
const cutoffIso = new Date(Date.now() - 3 * 60 * 60 * 1000).toISOString();

const { data } = await auth.supabase
  .from('messages')
  .select('id, sender_id, content, created_at')
  .gte('created_at', cutoffIso)
  .order('created_at', { ascending: true });

const recentMessages = ((data ?? []) as GlobalChatMessageRow[])
  .filter((message) => isMessageWithinThreeHours(message.created_at))
  .slice(-10);
~~~

Shared utility (app/api/_lib/mobile-rest.ts):
~~~ts
export function isMessageWithinThreeHours(createdAt: string): boolean {
  return new Date(createdAt).getTime() > Date.now() - 3 * 60 * 60 * 1000;
}
~~~

Mobile parity goal:
- Flutter must enforce all 4 layers:
  - Fetch cutoff at now - 3h.
  - Realtime ignore when message age exceeds 3h.
  - Keep in-memory list always truncated to last 10.
  - Apply final render-time age filter as defense-in-depth.
- If backend trigger exists, keep client filters anyway to prevent stale render race conditions.
